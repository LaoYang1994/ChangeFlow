# Agent Instructions — ChangeFlow plugin

This repo IS the ChangeFlow plugin, and it uses ChangeFlow on itself. The
durable docs below are this repo's own; the stubs under `templates/` are what
the plugin *ships* to other repos — don't confuse the two.

## Read order
Before starting work, read:
1. `docs/PROJECT.md` — what ChangeFlow is, the layout, and the design philosophy
2. `docs/CONCEPTS.md` if a term is unclear
3. `docs/CONTRACTS.md` before changing commands, scripts, or the doc structure
4. `docs/experiences/INDEX.md` to spot a relevant lesson, then read that file
5. `docs/workflows/INDEX.md` before a recurring task (adding a command, releasing,
   evaluating a prompt change) — follow the procedure if one matches

## Documentation system
Do not create random markdown files. Use only:
- `docs/PROJECT.md`, `docs/CONCEPTS.md`, `docs/CONTRACTS.md`
- `docs/workflows/` (procedures, with `INDEX.md`), `docs/experiences/` (lessons, with `INDEX.md`)
- `docs/changes/active/<YYYY-MM-DD>-<slug>/` and `docs/changes/archive/...`

## Change workflow
Non-trivial work is human-initiated via the `/change-*` commands this plugin
ships (design → plan → [optional plan-review] → implement → review → validate →
archive). Reference a change by its immutable ID, never by path.

## Entering implementation is human-triggered
Do not slide into the implement phase on your own. After design/plan — even if the
user approves the plan or asks for tweaks — do NOT start writing project code.
Revising `design.md`/`plan.md` per feedback is fine; writing code requires the user
to explicitly start implementation (run `/change-implement`), which is also where
the worktree choice is offered.

## Discuss before modifying
When the user is designing or deciding something, settle the design and get
explicit approval before editing files. A directional "yes" confirms the
*direction*, not approval to implement — especially when the change would reverse
a prior decision or a contract. Discuss to clarity first, then modify.

## Critical rules (full text in docs/CONTRACTS.md)
- **`commands/*.md` is the single source of truth.** `.codex/skills/` are
  GENERATED by `scripts/sync-codex.py` — never hand-edit them; regenerate with
  `--force`.
- **Command descriptions are trigger/"use when"-only**, never workflow summaries
  (the agent follows the blurb and skips the body otherwise). Bodies state each
  rule with its *why*. Keep them concise.
- **Per-change templates are embedded inside command bodies**, not referenced
  from `templates/`, because Codex skills can't reach the plugin root.
- **ChangeFlow is human-initiated**: commands never auto-create state; `change-*`
  Codex skills are marked explicit-invocation-only.
- **`init.sh` never overwrites existing files.**
- **`CLAUDE.md` imports `@AGENTS.md`**; don't duplicate.
- Run `claude plugin validate . --strict` before committing; it must pass.
- **Never push to the remote without explicit user permission.**
